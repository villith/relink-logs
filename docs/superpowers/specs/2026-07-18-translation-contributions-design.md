# Translation Contributions — Design

**Date:** 2026-07-18
**Status:** Approved pending final user review

## Goal

Let non-technical users (minimum requirement: a GitHub account) submit language/translation
changes, and make them aware of this from inside the app.

## Decisions made

- **Scope of contributions:** UI strings only — `src-tauri/lang/<lang>/ui.json`. All other
  lang files are autogenerated from game data and are overwritten on game updates; no
  overrides mechanism is added.
- **Submission mechanism:** GitHub **issue form only**. Contributors never edit files or
  open PRs; the maintainer applies reported fixes to `ui.json` by hand and closes the issue.
- **In-app awareness:** three right-aligned buttons in the Logs window header row
  (same row as the "GBFR Logs" title): **GitHub**, **Bugs**, **Translate**.
- **Target repo:** `https://github.com/villith/gbfr-logs` (the release fork the updater
  points at).

## Components

### 1. `.github/ISSUE_TEMPLATE/translation.yml`

GitHub issue form (structured web form; no markdown/JSON knowledge needed):

- Title prefix `[Translation]: `, label `translation` (label created in the repo via `gh`).
- **Language** — dropdown of the 10 supported languages (from `SUPPORTED_LANGUAGES` in
  `src/i18n.ts`) plus "Other / request a new language".
- **Type of change** — dropdown: wrong translation / typo / text shows in English
  (untranslated) / new language request.
- **Where it appears** — textarea; hint that screenshots can be pasted directly.
- **Current text** — input (optional for new-language requests).
- **Suggested text** — input.
- **Additional changes** — optional textarea so one issue can batch several fixes.

Note in the form description: item/weapon/skill *names* come from game data and cannot be
hand-fixed — only UI text is in scope.

### 2. `.github/ISSUE_TEMPLATE/bug.yml`

Simple bug-report form (target for the new Bugs button):

- Title prefix `[Bug]: `, label `bug`.
- What happened / steps to reproduce (textarea).
- App version, game version (inputs).
- Hint to attach the log file at `%APPDATA%\gbfr-logs\gbfr-logs.txt`.

### 3. Header buttons — `src/pages/Logs.tsx`

The `AppShell.Header` `Group` becomes `justify="space-between"`: burgers + "GBFR Logs"
title grouped left; a right-aligned `Group` of three compact subtle Mantine buttons:

| Button | Icon (`@phosphor-icons/react`) | Opens |
|---|---|---|
| GitHub | `GithubLogo` | `https://github.com/villith/gbfr-logs` |
| Bugs | `Bug` | `https://github.com/villith/gbfr-logs/issues/new?template=bug.yml` |
| Translate | `Translate` | `https://github.com/villith/gbfr-logs/issues/new?template=translation.yml` |

Links open in the system browser via `open` from `@tauri-apps/api/shell`
(`shell.open` is already allowlisted in `tauri.conf.json`; same pattern as
`src/utils.ts`).

Button labels are new i18n keys in `src-tauri/lang/en/ui.json`: `ui.github`,
`ui.report-bug`, `ui.help-translate` — so the notice itself is translatable and falls
back to English.

### 4. Documentation pointers

- `README.md`: short "Found a translation problem or bug?" section linking to the two
  issue forms.
- `src-tauri/lang/README.md`: add that non-technical contributors should use the
  translation issue form; direct edits remain welcome for those comfortable with PRs,
  still restricted to `ui.json`.

## Maintainer workflow

1. Issue arrives with the `translation` label.
2. Maintainer edits (or creates — several languages have no `ui.json` yet) the language's
   `ui.json` with the suggested text.
3. Commit references and closes the issue.

## Error handling / edge cases

- Language has no `ui.json` yet (e.g. `fr`, `jp`): maintainer creates the file on first
  fix; i18next already falls back to `en` for missing keys.
- Requested fix targets an autogenerated name file: out of scope; the form description
  states this and the maintainer closes with an explanation.
- `translation` / `bug` labels must exist in the repo or GitHub silently drops them from
  the form — created once via `gh label create`.

## Testing

- Frontend: `npm run build` (typecheck) and `npx vitest run` still pass; buttons render
  and `open` is called with the right URLs (manual check in `npm run dev`).
- Issue forms: validate YAML renders by opening the new-issue chooser on GitHub after
  push (GitHub is the only real validator).

## Out of scope

- Overrides mechanism for autogenerated lang files.
- Bot/Action automation converting issues into PRs.
- Crowdin/Weblate or any external translation platform.
